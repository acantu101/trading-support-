The Problem Before FIX
Early 1990s — every firm spoke a different language

The Problem

  • Every broker and exchange had custom proprietary formats
  • N brokers × M exchanges = N×M bespoke integrations
  • Phone calls, fax machines, custom binary protocols
  • Every format change broke every connection
  • Enormous cost and operational risk industry-wide

The Solution

  • FIX created in 1992 by Fidelity & Salomon Brothers
  • One shared language for all financial messaging
  • Open standard — not owned by any single firm
  • Adopted by virtually every broker, exchange, and asset manager worldwide
  • Processes trillions of USD in orders daily
Key insight: FIX was not the first protocol — it was the first one the entire industry agreed to adopt together.

FIX PROTOCOL · SLIDE 5 / 22
Message Structure: Header · Body · Trailer
Every FIX message has the same three-part structure

HEADER — Tags 8,9,35,49,56,52

  • BeginString — protocol version (8=FIX.4.4)
  • BodyLength — byte count of body
  • MsgType — what kind of message (35=D)
  • SenderCompID / TargetCompID
  • SendingTime — timestamp of send

BODY — payload, varies by MsgType

  • Symbol (55) — what to trade
  • Side (54) — buy or sell
  • OrderQty (38) — how many shares
  • Price (44) — at what price
  • OrdType (40) — limit, market, etc.

TRAILER — Tag 10 only

  • CheckSum — sum of all byte values mod 256
  • Receiver validates this first
  • Corrupt message → reject immediately
  • Always the very final field
Analogy: Header = envelope (sender, recipient, date) · Body = the letter · Trailer = wax seal proving no tampering

FIX PROTOCOL · SLIDE 7 / 22
FIX Session Lifecycle
Before any trade messages flow, a session must be established

1. Establish

  • Client sends Logon (A)
  • Broker echoes Logon (A)
  • Session is live

2. Active

  • Heartbeat (0) every N seconds
  • Test Request (1) if missed
  • Missed heartbeat → session drops

3. Trading

  • All message types flow
  • Orders, reports, cancels, replaces

4. Close

  • Either side sends Logout (5)
  • Graceful close preserves sequence state
Sequence numbers: Every message gets an incrementing MsgSeqNum (tag 34). Gap detected → ResendRequest (MsgType 2). Resets only on Logon with ResetSeqNumFlag=Y.
Golden rule: Never just drop the TCP connection. Always send Logout (5) first — graceful close preserves sequence state.

Which FIX Version?
The safe default for new integrations is FIX 4.4

FIX 4.2 — 1998

  • Equities workhorse
  • Simple, well-understood
  • Limited to equity asset class
  • Many legacy systems still on 4.2

FIX 4.4 — 2003 ★ USE THIS

  • Most common version today
  • Derivatives, FX, fixed income
  • Safe default for new integrations
  • Broad counterparty support

FIX 5.0 / FIXT 1.1 — 2006

  • Separated session from app layer
  • Allows non-FIX transports
  • More complex to implement
  • Less universally adopted
Practical advice: Use whatever version your counterpart specifies. When in doubt, FIX 4.4 is the safe default.

What you now know about FIX

  • Tag=value pairs — every message is a flat list of numbered fields; Tag 35 (MsgType) tells you what kind of message it is

ITCH PROTOCOL · SLIDE 11 / 22
ITCH Protocol — Overview
NASDAQ's outbound market data protocol — what you receive from the exchange

What it is

  • Binary, one-way protocol — exchange sends, you receive
  • Delivers full order book updates in real time
  • Every add, modify, cancel, and execution on NASDAQ
  • Used by feed handlers to build and maintain L2 order books
  • Delivered over UDP multicast (not TCP)

Why binary, not text?

  • Much smaller message size — less network bandwidth
  • Faster to parse — no ASCII conversion, direct memory copy
  • Lower latency — critical for HFT market data
  • Trade-off: not human-readable in logs
  • Support engineers read decoded/logged output, not raw bytes
Support context: You won't read raw ITCH bytes. Your feed handler decodes them and logs structured output. You read the logs.

OUCH PROTOCOL · SLIDE 15 / 22
OUCH Protocol — Overview
NASDAQ's binary order entry protocol — what you send to the exchange

What it is

  • Binary, two-way protocol — client sends orders, exchange responds
  • NASDAQ's proprietary alternative to FIX for order entry
  • Lower latency than FIX — binary parsing vs ASCII
  • Delivered over TCP (reliable, ordered) — unlike ITCH
  • Used by HFT firms that need microsecond-level order entry

OUCH vs FIX

  • OUCH: binary, NASDAQ-specific, ultra-low latency
  • FIX: text, universal, easier to integrate and debug
  • Both accomplish the same goal: submit/manage orders
  • Most HFT firms use OUCH; most institutional firms use FIX
  • As support engineer: you'll see OUCH in logs but won't hand-craft messages
Support context: If an order went missing or was rejected, you'll look at OUCH session logs — decoded by the OMS into structured lines, similar to FIX logs.

BINARY / NATIVE · SLIDE 18 / 22
Why Binary Protocols Exist
FIX is text. ITCH and OUCH are binary. Here is why it matters.

Text protocols (FIX)

  • Human-readable in logs — easy to debug
  • Larger message size (ASCII chars vs packed bytes)
  • Slower to parse — must scan for delimiter, convert ASCII to numbers
  • Order of tags not guaranteed
  • Good for: institutional order management, broker connectivity

Binary protocols (ITCH, OUCH, SBE)

  • Not human-readable — must be decoded by software
  • Much smaller message size — less bandwidth, less CPU
  • Faster to parse — fixed offsets, no delimiter scanning
  • Deterministic layout — field at byte N, always
  • Good for: HFT market data, ultra-low-latency order entry
SBE (Simple Binary Encoding): FIX Trading Community's official binary encoding standard. Used where FIX semantics (tags, message types) are needed but text overhead is too high. CME, LSE, and others use SBE.

A/B Feed Redundancy

  • Primary (A) and Secondary (B) carry identical data
  • Two separate network paths for resilience
  • Use whichever arrives first; fall back to other on gap
  • Both must be subscribed (joined via IGMP)

UDP vs TCP tradeoffs

  • No TCP handshake = lower latency delivery
  • No retransmission = dropped packet = lost tick
  • Must detect gaps via sequence numbers
  • Gap fill requested via separate TCP channel

Multicast Diagnostics — rmem_max & IGMP
The two most common multicast support issues
# 1. Check if host has joined the multicast group (IGMP membership)
netstat -g
ip maddr show eth0
# If multicast group 233.54.12.100 is NOT listed → feed handler never joined
# Fix: restart feed handler; check socket join call; check network switch IGMP snooping

# 2. Check socket receive buffer (rmem_max)
cat /proc/sys/net/core/rmem_max
# Default ~212992 (208KB) — too small for burst market data at open
# If feed handler logs "socket buffer overflow" or drops spikes at 09:30 → raise this
sysctl -w net.core.rmem_max=33554432   # 32MB — common HFT setting
sysctl -w net.core.rmem_default=33554432

# 3. Verify packets are arriving on the interface
tcpdump -i eth0 -nn host 233.54.12.100 -c 20
# No packets → IGMP or network issue upstream; packets present → app-level issue
Symptom pattern: Gaps spike exactly at market open (09:30 ET) then clear — almost always rmem_max too small. The burst of adds at open overflows the socket buffer.

KAFKA · SLIDE 1 / 8
Kafka Architecture Overview
Distributed messaging backbone — every trade event, risk update, and market data tick flows through it

Core Components

  • Broker — Kafka server storing and serving messages; cluster = multiple brokers for redundancy
  • Topic — named stream (e.g. "trade-executions", "market-data", "risk-updates")
  • Partition — topics split for parallelism; each partition is strictly ordered
  • Offset — record's position within a partition; consumers track this to know where they left off

Producers & Consumers

  • Producer — writes messages to a topic; chooses partition (by key hash or round-robin)
  • Consumer Group — one or more consumers sharing a group-id; each partition assigned to exactly ONE consumer
  • Replication — each partition has a leader + replica copies; RF=3 is the HFT standard
  • ISR — In-Sync Replicas; only ISR members can become leader

Consumer Groups & Lag
Lag is your primary operational signal — it tells you if consumers are keeping up
  • Consumer lag = LOG-END-OFFSET − CURRENT-OFFSET per partition; high lag = consumer falling behind
  • CONSUMER-ID showing "-" = no consumer assigned to that partition → the pod has died
  • Same cutoff timestamp across multiple symbols = pipeline crash (not per-symbol data quality issue)
  • Lag growing on all partitions simultaneously = consumer failure, not a producer or broker issue
  • Lag on one partition = that partition's consumer is slow or dead; check pod restarts
# Describe consumer group lag
kafka-consumer-groups.sh --bootstrap-server broker:9092 \
  --describe --group tick-storage-group

# Output columns:
# TOPIC        PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG  CONSUMER-ID
# market-data  0          10042           10042           0    tick-pod-1
# market-data  1          8891            12778           3887 -          ← dead
CONSUMER-ID "-" on all partitions means every consumer in the group is gone — the entire pod fleet is down. This is K-12 territory: check kubectl logs --previous.

KAFKA · SLIDE 4 / 8
Consumer Config & Offset Management
The most dangerous default in Kafka is enable.auto.commit=true
# WRONG — auto-commit commits offset BEFORE processing is confirmed
# crash after commit but before write → data silently lost
consumer = KafkaConsumer(
    'market-data',
    enable_auto_commit=True,   # ← DANGEROUS
)

# RIGHT — commit only after successful write to HDF5
consumer = KafkaConsumer(
    'market-data',
    enable_auto_commit=False,  # ← always
    auto_offset_reset='earliest',  # replay from start on new group
    max_poll_records=500,         # limit batch; too large = timeout → rebalance
)
for msg in consumer:
    write_to_hdf5(msg.value)    # write FIRST
    consumer.commit()           # commit AFTER successful write

auto.offset.reset values

  • earliest — replay from beginning; use for new consumer groups or data recovery
  • latest — skip old messages; only for non-critical consumers

Kafka retention window

  • How long messages are kept on broker (e.g. 7 days)
  • After expiry, replay is impossible — urgency flag for L2 escalation

Interview answer: "We commit after write, so at-least-once. We handle potential duplicates on the write side using the tick's sequence number as an idempotency key — a replay can't insert the same tick twice."

Workload Objects

  • Pod — smallest deployable unit; one or more containers sharing network + storage
  • Deployment — desired state declaration; manages rolling updates and replica count
  • Job — runs a pod to completion; used for one-off tasks
  • CronJob — scheduled Jobs; K8s-native cron

Config & Network

  • Namespace — virtual cluster for isolation (trading, market-data, monitoring)
  • Service — stable network endpoint to reach pods regardless of pod IP changes
  • ConfigMap / Secret — externalize configuration and credentials from images
  • Resource requests/limits — requests = guaranteed; limits = hard ceiling; exceed = OOMKilled or throttled
Interview angle: "In trading, a bad pod restart during market hours means researcher data gaps or live trader impact depending on the workload — so we treat pod health as P1."

ArgoCD vs Argo Workflows
Same project, completely different products — a common interview trip wire

  • Watches a Git repo for K8s manifest changes
  • Syncs the cluster state to match Git on every commit
  • Deploys and manages applications (Deployments, Services, etc.)

Argo Workflows — Job Orchestration

  • Like K8s-native Airflow — steps, DAGs, templates

  • Supports CronWorkflow for scheduled pipelines

Argo Workflow YAML Anatomy
What a workflow definition looks like and where to look for resource problems
apiVersion: argoproj.io/v1alpha1
kind: Workflow
metadata:
  name: tick-storage-workflow
spec:
  entrypoint: main            # which template to run first
  templates:
  - name: main
    steps:                    # sequential steps (use dag: for parallel)
    - - name: fetch-ticks
        template: fetch-ticks-tmpl
    - - name: write-hdf5       # runs after fetch-ticks completes
        template: write-hdf5-tmpl

  - name: write-hdf5-tmpl
    container:
      image: tick-storage:latest
      resources:
        requests: { memory: "512Mi", cpu: "500m" }
        limits:   { memory: "1Gi",  cpu: "1" }  # OOMKilled if exceeded
Where to look: resources.limits.memory is the first thing to check when a step OOMKills. The limit is a hard ceiling — the kernel kills the container the instant it exceeds it.

K8-08 — retry vs resubmit
Two different recovery commands with very different behavior

argo retry <workflow>

  • Resumes from the failed step only
  • Reuses outputs from all completed steps
  • Faster — skips work already done
  • Use flag --restart-successful to preserve completed steps
  • Same workflow name, same run ID

argo resubmit <workflow>

  • Starts from scratch — new workflow entirely
  • New workflow name, fresh run ID
  • All steps re-run from the beginning
  • Slower — re-fetches all data
  • Use when data or inputs may be corrupt

K8-09 — CronWorkflow Missed Run
Silent failure: concurrencyPolicy: Forbid + stuck workflow = skipped runs with no alert
# CronWorkflow definition — the trap is concurrencyPolicy: Forbid

  schedule: "30 16 * * 1-5"        # run at 16:30 Mon-Fri

L1 / L2 / L3 Order Book
What researchers mean when they say "the book looks wrong"

L1 — Best Quote (NBBO)

  • Best bid price + size
  • Best ask price + size
  • National Best Bid and Offer
  • What retail brokers show
  • Derived from L2 top of book

L2 — Market Depth

  • Full book by price level (aggregated)
  • Shows liquidity depth at each level
  • What algo traders use for sizing
  • Built from ITCH Add/Delete/Execute msgs
  • Gap → book is stale → must invalidate

L3 — Full Order Book

  • Individual order visibility
  • Every add/cancel/execute per order
  • Only some venues provide L3
  • Used for analytics, not live trading
  • Highest data volume
Stale book trigger: Any ITCH sequence gap means you missed Add/Delete/Execute messages. The book now shows wrong prices or phantom orders. Must invalidate and rebuild from snapshot.
MARKET DATA · SLIDE 3 / 8
Feed Handler — What It Does
The process that sits between the exchange and your infrastructure
  • Subscribes to UDP multicast groups (IGMP join) for each exchange feed
  • Parses raw binary ITCH/SBE messages into structured events
  • Sequences messages and detects gaps (expected N+1, received N+5 = gap)
  • Maintains an in-memory order book per symbol, applying Add/Delete/Execute in order
  • Publishes normalised tick data to Kafka for downstream consumers
  • Requests gap fills or snapshots via TCP when UDP gaps are detected

Market Impact & Slippage
What researchers mean when they say "fills look off"
  • Slippage = (actual fill price − expected price) / expected price × 10,000 basis points
  • 1 bp = 0.01% = $0.01 per $100 notional — small per unit, large at scale
  • Market impact: a large order moves the market against you — buying 1M shares pushes the ask up
  • VWAP benchmark: was the fill above or below VWAP for the execution period?
# VWAP = Volume-Weighted Average Price
# If you filled at $150.05 and VWAP was $149.90:
slippage = (150.05 - 149.90) / 149.90 * 10000
# = 10.0 basis points — meaningful for a large order

# Support triage when researcher says "fills look off":
# 1. Pull tick data for the execution window from HDF5
# 2. Calculate VWAP over the execution period
# 3. Compare execution price to VWAP
# 4. Check if PRIMARY vs SECONDARY feed data matches (slide 8)
# 5. Check for gaps in feed during execution window
Interview angle: "Slippage analysis requires accurate tick data. If the feed had a gap during the execution window, VWAP is wrong — so the slippage number is wrong too. We check data quality before interpreting analytics."

tick_data Schema
The core market data table you will query in every support investigation
CREATE TABLE tick_data (
    id       INTEGER PRIMARY KEY,
    symbol   TEXT    NOT NULL,
    ts       TEXT    NOT NULL,      -- '2024-01-15 14:30:00.123'
    price    REAL    NOT NULL,
    volume   INTEGER NOT NULL,
    seq_num  INTEGER NOT NULL,      -- monotonically increasing per (symbol, feed)
    feed     TEXT    NOT NULL CHECK(feed IN ('PRIMARY', 'SECONDARY'))
);
CREATE INDEX idx_tick_symbol_ts   ON tick_data(symbol, ts);
CREATE INDEX idx_tick_feed_symbol ON tick_data(feed, symbol, ts);

Key columns

  • feed — PRIMARY = exchange A feed, SECONDARY = exchange B backup
  • seq_num — monotonically increasing per (symbol, feed); gap = missed ITCH messages
  • ts — always filter on this with indexed columns for performance

Index strategy

  • (symbol, ts) — for range queries by symbol over time
  • (feed, symbol, ts) — for feed-specific queries; feed first narrows scan fast
  • Always filter on indexed columns before window functions

Zero-Tick Window Detection
Finding symbols with no data in a time window — the first query you run on a gap ticket
-- Which symbols have NO ticks in 14:30-15:00 on PRIMARY feed?
-- EXCEPT: symbols in the full table but absent in the window
SELECT DISTINCT symbol FROM tick_data
EXCEPT
SELECT DISTINCT symbol FROM tick_data
WHERE  feed = 'PRIMARY'
  AND  ts  >= '2024-01-15 14:30:00'
  AND  ts  <  '2024-01-15 15:00:00';

-- Count ticks per symbol in window (0 = gap, low = partial)
SELECT symbol, COUNT(*) AS tick_count
FROM   tick_data
WHERE  feed = 'PRIMARY'
  AND  ts >= '2024-01-15 14:30:00'
  AND  ts <  '2024-01-15 15:00:00'
GROUP BY symbol
ORDER BY tick_count;   -- zeros and low counts first
Pattern: EXCEPT is your zero-tick finder. COUNT with ORDER BY ASC shows you the worst-affected symbols first — useful when a partial outage hit some symbols more than others.

Sequence Gap Detection with LAG()
Finding where seq_num jumps by more than 1 — confirms ITCH messages were missed
-- Find sequence gaps: where seq_num jumps by more than 1
SELECT symbol, feed, seq_num, prev_seq,
       seq_num - prev_seq - 1 AS missing_count
FROM (
  SELECT symbol, feed, seq_num,
         LAG(seq_num) OVER (
           PARTITION BY symbol, feed
           ORDER BY seq_num
         ) AS prev_seq
  FROM tick_data
) sub
WHERE prev_seq IS NOT NULL
  AND seq_num - prev_seq > 1
ORDER BY missing_count DESC;
  • LAG(seq_num) returns the previous row's seq_num within the (symbol, feed) partition
  • seq_num − prev_seq − 1 = how many messages are missing between these two rows
  • PARTITION BY symbol, feed — critical; without it, LAG crosses symbol boundaries giving false gaps
  • First row per (symbol, feed) has prev_seq IS NULL — exclude it with the WHERE clause

Feed Completeness Cross-Check
Finding ticks present on PRIMARY but missing on SECONDARY (or vice versa)
-- Ticks on PRIMARY with no matching SECONDARY tick
-- LEFT JOIN + IS NULL = "exists on left, missing on right"
SELECT p.symbol, p.ts, p.price AS primary_price
FROM   tick_data p
LEFT JOIN tick_data s
  ON  s.symbol = p.symbol
  AND s.ts     = p.ts
  AND s.feed   = 'SECONDARY'
WHERE  p.feed = 'PRIMARY'
  AND  p.ts >= '2024-01-15 14:30:00'
  AND  p.ts <  '2024-01-15 15:00:00'
  AND  s.id IS NULL          -- no matching SECONDARY row
ORDER BY p.symbol, p.ts;

What this tells you

  • PRIMARY had ticks at these timestamps that SECONDARY never received
  • SECONDARY feed had a dropout — possible network issue on B path
  • Swap feeds: check if SECONDARY has ticks PRIMARY missed

When to use

  • Researcher asks: "is the backup feed reliable?"
  • Post-incident: was the secondary feed good enough to replay from?
  • Both feeds gapped at same time = exchange issue, not internal

Window Functions — OVER / PARTITION BY
The most important SQL feature for time-series market data analysis
-- OVER clause: apply aggregate across rows WITHOUT collapsing them

-- Row number per symbol (useful for pagination / sampling)
ROW_NUMBER() OVER (PARTITION BY symbol ORDER BY ts)

-- Previous row's seq_num — gap detection
LAG(seq_num, 1) OVER (PARTITION BY symbol, feed ORDER BY seq_num)

-- Next row's value
LEAD(price, 1)  OVER (PARTITION BY symbol ORDER BY ts)

-- Running volume total per symbol
SUM(volume) OVER (
  PARTITION BY symbol
  ORDER BY ts
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)

-- 10-tick moving average price
AVG(price) OVER (
  PARTITION BY symbol
  ORDER BY ts
  ROWS BETWEEN 9 PRECEDING AND CURRENT ROW
)

VWAP Query
Volume-Weighted Average Price — the standard fill quality benchmark
-- VWAP per symbol for a full session
SELECT
    symbol,
    ROUND(SUM(price * volume) * 1.0 / SUM(volume), 4) AS vwap,
    COUNT(*)       AS tick_count,
    MIN(price)     AS low,
    MAX(price)     AS high,
    SUM(volume)    AS total_volume
FROM   tick_data
WHERE  feed = 'PRIMARY'
  AND  ts >= '2024-01-15 09:30:00'
  AND  ts <  '2024-01-15 16:00:00'
GROUP BY symbol
ORDER BY total_volume DESC;
  • VWAP = Σ(price × volume) / Σ(volume) — weights each price by how much traded at that level
  • Use * 1.0 to force float division in SQLite (avoids integer truncation)
  • Compare a trader's fill price to VWAP: fill above VWAP (buy) = slippage; below = good execution
  • If there are gaps in tick_data, VWAP is wrong — always check data quality before interpreting

Sensor Anti-patterns
poke mode is the most common cause of Airflow worker starvation

WRONG — poke mode

SqlSensor(
    task_id='wait_for_data',
    mode='poke',          # holds worker slot
    poke_interval=30,    # polls every 30s
    # no timeout set → hangs forever
)
  • Holds a worker slot while sleeping between polls
  • 10 sensors = 10 workers blocked = other DAGs starve
  • No timeout = sensor hangs forever on bad condition

RIGHT — reschedule mode

P-03 — Python Process Watchdog
Probe a critical process via its PID file and restart it before the trading desk notices
import subprocess, time, logging, os
from pathlib import Path

log = logging.getLogger(__name__)
logging.basicConfig(level=logging.INFO,
    format='%(asctime)s %(levelname)s %(message)s')

PID_FILE     = Path('/var/run/oms-gateway.pid')
CMD          = ['/opt/oms/bin/gateway', '--config', '/etc/oms/gateway.yml']
MAX_RESTARTS = 5

def is_running(pid: int) -> bool:
    try:
        os.kill(pid, 0)   # signal 0 = probe only, does not kill
        return True
    except (ProcessLookupError, PermissionError):
        return False

def start_process() -> int:
    proc = subprocess.Popen(CMD)
    PID_FILE.write_text(str(proc.pid))
    log.info("Started gateway PID %d", proc.pid)
    return proc.pid

restarts = 0
while restarts < MAX_RESTARTS:
    pid = int(PID_FILE.read_text()) if PID_FILE.exists() else 0
    if not is_running(pid):
        log.warning("Gateway down — restarting (attempt %d/%d)",
                    restarts + 1, MAX_RESTARTS)
        pid = start_process()
        restarts += 1
    time.sleep(10)
Cap restarts. Without MAX_RESTARTS a misconfigured binary crashes immediately and the watchdog burns CPU in a tight loop. After N failures, page a human instead.

def get_with_retry(url: str, retries: int = 3, backoff: float = 2.0) -> Any:

from typing import Iterator

SOH = '\x01'   # FIX field delimiter

TAGS = {'8': 'BeginString', '35': 'MsgType',  '34': 'MsgSeqNum',
        '49': 'SenderCompID',              '56': 'TargetCompID',
        '7':  'BeginSeqNo',  '16': 'EndSeqNo',
        '36': 'NewSeqNo',   '43': 'PossDupFlag'}
MSG_TYPES = {'D': 'NewOrderSingle', '8': 'ExecutionReport',
             '2': 'ResendRequest',   '4': 'SequenceReset'}

def parse_fix(raw: str) -> dict:
    return {k: v for part in raw.split(SOH) if '=' in part
            for k, v in [part.split('=', 1)]}

    with open(log_path) as fh:
        for line in fh:

            seq = int(fields.get('34', 0))
            mt  = MSG_TYPES.get(fields.get('35', ''), 'Unknown')
            if seq > 0 and seq != expected:
                yield (expected, seq, mt)   # (expected, got, msgtype)
            if seq > 0:
                expected = seq + 1

P-07 — OOP: Strategy Pattern
Swap pricing algorithms at runtime without touching caller code — canonical trading-desk pattern
from abc import ABC, abstractmethod
from dataclasses import dataclass

@dataclass
class Tick:
    price: float; volume: int

class PricingStrategy(ABC):
    @abstractmethod
    def compute(self, ticks: list) -> float: ...

class VWAPStrategy(PricingStrategy):
    def compute(self, ticks) -> float:
        return (sum(t.price * t.volume for t in ticks)
                / sum(t.volume for t in ticks))

class MidpointStrategy(PricingStrategy):
    def compute(self, ticks) -> float:
        prices = [t.price for t in ticks]
        return (min(prices) + max(prices)) / 2

class LastPriceStrategy(PricingStrategy):
    def compute(self, ticks) -> float:
        return ticks[-1].price

class PriceEngine:
    def __init__(self, strategy: PricingStrategy):
        self._strategy = strategy

    def set_strategy(self, strategy: PricingStrategy):
        self._strategy = strategy   # hot-swap without restart

    def price(self, symbol: str, ticks: list) -> float:
        result = self._strategy.compute(ticks)
        print(f"[{symbol}] {type(self._strategy).__name__} → {result:.4f}")
        return result

P-08 — OOP: Observer Pattern
Fan out market data ticks to multiple subscribers — decouple the feed from every consumer
from abc import ABC, abstractmethod
from dataclasses import dataclass, field
from typing import List
import logging

@dataclass
class Tick:
    symbol: str; price: float; volume: int

class TickObserver(ABC):
    @abstractmethod
    def on_tick(self, tick: Tick) -> None: ...

class HDF5Writer(TickObserver):
    def on_tick(self, tick: Tick):
        print(f"HDF5 ← {tick.symbol} {tick.price}")

class RiskMonitor(TickObserver):
    def on_tick(self, tick: Tick):
        if tick.price <= 0:
            logging.error("Zero/negative price for %s", tick.symbol)

class MarketDataFeed:
    def __init__(self):
        self._observers: List[TickObserver] = []

    def subscribe(self, obs: TickObserver):   self._observers.append(obs)
    def unsubscribe(self, obs: TickObserver): self._observers.remove(obs)

    def publish(self, tick: Tick):
        for obs in self._observers:
            try:
                obs.on_tick(tick)
            except Exception as e:
                logging.error("Observer %s failed: %s", type(obs).__name__, e)
Isolate observer failures. The try/except inside publish() is intentional: a broken HDF5Writer (disk full) must not prevent RiskMonitor from receiving ticks. Never let one subscriber take down the bus.

VPC & Security Groups

VPC Design for Trading

  • Public subnets — load balancers, NAT gateways only
  • Private subnets — all application workloads (EKS nodes, Kafka, DBs)
  • Isolated subnets — RDS, ElastiCache; no route to internet
  • One VPC per environment (prod / staging / dev); VPC peering or Transit Gateway for shared services
CIDR planning matters. EKS with VPC CNI assigns one IP per pod — undersized subnets cause pod scheduling failures.

TCP/IP Essentials for Trading Support

3-Way Handshake

Client SYN→ Server SYN-ACK→ Client ACK→ Connected
  • TIME_WAIT — closed connection held 2×MSL (60s). High-frequency systems exhaust ephemeral ports; tune net.ipv4.tcp_tw_reuse=1
  • FIN_WAIT_2 — remote closed but local hasn't; watch for FIX session teardown hangs
  • CLOSE_WAIT — local hasn't called close(); usually an app bug (not reading EOF)
ss -s showing thousands of TIME_WAIT → ephemeral port exhaustion risk. Check cat /proc/sys/net/ipv4/ip_local_port_range.

Key TCP Tuning Parameters

# View current values
sysctl net.core.rmem_max
sysctl net.core.wmem_max
sysctl net.ipv4.tcp_rmem   # min default max

# Trading system tunings
net.core.rmem_max = 134217728    # 128MB
net.core.wmem_max = 134217728
net.ipv4.tcp_rmem = 4096 87380 134217728
net.ipv4.tcp_wmem = 4096 65536 134217728
net.ipv4.tcp_no_metrics_save = 1  # don't cache old RTT
Nagle's Algorithm: TCP buffers small writes for efficiency — terrible for order entry latency. Disable with TCP_NODELAY socket option on FIX/OUCH connections.

UDP Multicast — Market Data Delivery

How Multicast Works

  • Exchange sends one UDP stream; many receivers join via IGMP
  • Multicast address range: 224.0.0.0/4
  • Switches use IGMP snooping to forward only to ports that joined
  • No delivery guarantee — gaps are normal; application must detect and recover
Exchange→ UDP multicast 224.x.x.x:PORT→ Feed Handler (joined via IGMP)
A/B feeds are identical streams on different multicast groups and NICs. Feed handler merges them by sequence number — whichever arrives first wins.

Joining & Diagnosing

As a support engineer you won't configure these, but you'll see references in feed handler configs (ef_vi, onload prefix in logs). Know they exist and that packet drops there need HW-level investigation.

git bisect — Hunting Regressions

What It Does

Binary searches commit history to find the exact commit that introduced a bug. Halves the search space each step — 1000 commits = ~10 steps.

# Start bisect
git bisect start

# Mark current state as broken
git bisect bad

# Mark a known-good commit (tag or hash)
git bisect good v2.1.0

# Git checks out the midpoint commit.
# Test it, then tell git the result:
git bisect good   # or: git bisect bad

# Repeat until git prints:
# "abc1234 is the first bad commit"

# Done — clean up
git bisect reset

Automated Bisect

# Script returns 0 = good, non-zero = bad
git bisect run python tests/check_lag.py

# Example check script
#!/usr/bin/env python3
import subprocess, sys
result = subprocess.run(
    ['python', '-m', 'pytest', 'tests/test_consumer.py', '-x', '-q'],
    capture_output=True
)
sys.exit(result.returncode)
Trading use case: "Sequence gap rate doubled after last deploy" — bisect between last release tag and HEAD with a script that replays a captured feed and counts gaps.
Bisect only works cleanly if you can reliably test each commit. Build failures at midpoints = use git bisect skip.

Branching Strategy for Trading Systems

Common Model (trunk-based)

main — always deployable; protected; requires PR + review
feature/xyz — short-lived; off main; merged via PR
release/v2.3 — cut from main; only bug fixes cherry-picked in
hotfix/critical-gap — off release branch; merged back to both release AND main
Long-lived feature branches are merge debt. In fast-moving trading systems, divergence causes painful conflicts at release time. Prefer small, frequent merges.

Key Commands

# Cut release branch
git checkout main
git checkout -b release/v2.3
git push -u origin release/v2.3

# Cherry-pick a fix onto release
git checkout release/v2.3
git cherry-pick abc1234   # specific commit from main

# Tag the release
git tag -a v2.3.1 -m "Fix sequence gap detection"
git push origin v2.3.1

# Compare release to main
git log main..release/v2.3 --oneline
git diff main...release/v2.3
git diff A...B (three dots) shows what B added since it diverged from A — useful for PR reviews and release notes.

First 5 Minutes Checklist

  • ✅ Acknowledge the alert — claim ownership so no one duplicates work
  • ✅ Assess blast radius — how many systems/users affected?

Postmortem

Mitigation vs Resolution

  • Mitigation — stops the bleeding; system functional again. May be a workaround (restart, rollback, redirect traffic).
  • Resolution — root cause fixed; no workaround needed. May come days later.
  • Close the incident on mitigation, not resolution — trading can resume. Track the root-cause fix as a separate ticket.
Rollback is almost always the fastest mitigation. If a deploy triggered the incident, revert first and investigate second.

Roles in a P1

  • Incident Commander (IC) — owns the bridge; delegates; makes go/no-go calls. Not debugging — managing.
  • Tech Lead — does the hands-on diagnosis and fix. Reports findings to IC.
  • Comms Lead — writes status updates; keeps stakeholders informed so IC/Tech Lead aren't interrupted.
  • Scribe — timestamps every action in the doc. Often the IC in smaller teams.
Small teams collapse roles. At minimum: one person owns comms, one person owns technical work. Don't let one engineer debug AND answer questions simultaneously.

Trading-Specific Triage Patterns

Feed Gap / Data Missing

  1. Check feed handler logs for sequence gap messages
  2. Is it one symbol or all? → symbol = upstream issue; all = connectivity/process
  3. Is A feed or B feed also gapped? → both = upstream or NIC; one = network path
  4. Check Kafka consumer lag — is data in Kafka but not consumed?
  5. Check HDF5 file shape vs expected row count for the day
During market hours, a data gap is a P1 until proven otherwise. Researchers and algos may be making decisions on stale data right now.

Order Not Reaching Exchange

  1. Check FIX session state — is it logged on? (35=A in logs)
  2. Check Execution Report (35=8) for reject reason (tag 103)
  3. Check OUCH Rejected message — RejectReason byte tells you exactly why
  4. Check sequence numbers — gap = session will reject until reset
  5. Check port/network connectivity to exchange gateway

Elevated Latency

  1. kubectl top pod — CPU/memory pressure?
  2. ss -s — socket buffer overflow? TIME_WAIT pile-up?
  3. GC pause? Check JVM logs if Java-based components involved
  4. Noisy neighbor? Check node-level CPU steal with top

Escalation Ladder

  • Self-resolve (15 min) — you have a runbook, clear cause, known fix
  • Escalate to senior (30 min) — unclear root cause or fix requires access you don't have
  • Escalate to vendor (P1) — exchange connectivity, MSK broker failure, AWS service issue
  • Escalate to management — financial impact, regulatory implication, breach of SLA
Never sit on an incident. If you've been stuck for 20 minutes without progress, escalate. The cost of over-escalating is low; the cost of under-escalating during market hours is high.

"I have a P1. [System] is [symptom] since [time]. I've ruled out [X] and [Y]. I need help with [specific thing]. Can you join the bridge?"

Root Cause Analysis

5 Whys — Work Backward

Problem: Researchers got stale data for 40 minutes
Why 1: HDF5 writer pod was OOMKilled and restarted
Why 2: Memory limit was 512Mi but dataset grew 3x this quarter
Why 3: Resource limits were set at launch and never reviewed
Why 4: No memory growth alerting was configured
Root cause: No process for reviewing resource limits as workloads scale

Timeline Reconstruction

  • Pull all relevant logs with a common time window (start 30 min before symptom)
  • Correlate: app logs + k8s events + monitoring alerts + deploy history
  • Find the earliest anomaly — that's your candidate root cause
# Pull k8s events sorted by time
kubectl get events -n trading \
  --sort-by='.lastTimestamp' | tail -50

# Correlate pod restart time with app error
kubectl logs tick-writer --previous \
  | grep -E "ERROR|WARN|OOM" | tail -30
The root cause is almost never "it just crashed." Look for the trigger: a deploy, a config change, a data volume threshold crossed, a dependency that became unavailable.

Runbook Best Practices

What Makes a Good Runbook

  • Written for 3am — assume the reader is tired, stressed, and unfamiliar with this system
  • Symptom-first — index by what alerts fire or what the user reports, not by system name
  • Copy-pasteable commands — no "run the usual check"; give the exact command with real variable names
  • Decision tree, not prose — "if X → go to step 4; if Y → go to step 7"
  • Escape hatch — always end with "if none of the above worked, escalate to [person/team]"
  • Dated — include last-verified date; stale runbooks are dangerous

Blameless Postmortem

Principles

  • Blameless — people make reasonable decisions given the information they had. The system allowed the failure; fix the system.
  • Focus on contributing factors, not the person who "caused" it
  • Action items must be assigned and dated — unowned items never get done
  • Share widely — postmortems are a knowledge transfer mechanism, not a shame exercise
A good postmortem surfaces 3–5 contributing factors and generates 3–5 action items. If you only have one cause and one fix, you probably stopped asking "why" too early.

Order Routing

  • FIX 4.2 / 4.4 / 5.0 — standard for ATS connectivity
  • OUCH (binary) — Nasdaq-style low-latency alternative

  • TCP port/IP whitelist — firewall change = silent drops
  • TLS cert expiry — ATS rejects unencrypted connections

Key Concepts

Regulatory Framework & Support Tickets
What governs dark pools — and the tickets that land in the support queue

Key Regulations

  • Reg ATS (1998): ATS must register with SEC if daily volume > 5% of national volume in a security
  • Rule 606: Brokers must quarterly disclose order routing venues and payment arrangements
  • FINRA Rule 4552: ATS reports biweekly share volume per symbol — published publicly
  • Rule 15c3-5 (Market Access): Pre-trade risk controls required before routing to any ATS
  • Reg NMS Rule 611: Trade-through protection — dark fills must not be at worse price than NBBO

Common Support Tickets

  • TRF submission failure (PRICE_OUTSIDE_NBBO, session drop)
  • FIX session reset — sequence mismatch at SOD causing reject storm
  • IOI not reaching counterparty — network or config issue
  • Fill rate drop — SOR config change or dark pool capacity issue
  • Execution quality complaint — TCA pull + adverse selection analysis
  • Late TRF report — breach of 10-second reporting obligation
Escalation path: ATS ops → FINRA Market Regulation (TRF issues) → SEC (systematic ATS violations). Always document the timeline and error codes before escalating.
